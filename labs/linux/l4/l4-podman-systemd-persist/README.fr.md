# Lab — conteneur persistant au boot avec Quadlet

## Rappel

[**Quadlet sur le guide compagnon**](https://blog.stephane-robert.info/docs/conteneurs/moteurs-conteneurs/podman/quadlet/)

Quadlet transforme les fichiers `*.container` déposés dans un répertoire connu
de systemd en services, au `daemon-reload`. Une section `[Container]` décrit
l'image et la commande ; la section `[Install] WantedBy=` fait démarrer le
service généré au boot. Le nom du service est celui du fichier, suivi de
`.service`.

## Le cours

Les exemples ci-dessous montent un service nommé `veilleuse`, en **rootless**,
à partir de l'image `docker.io/library/alpine:3.21` : le challenge, lui,
demandera un autre nom, une autre image et l'autre mode. Le but est d'apprendre
le geste et de savoir le transposer, pas de recopier une ligne. Toutes les
sorties viennent d'une VM **AlmaLinux 10.2** en **Podman 5.8.2**, utilisateur
`ansible` d'**UID 1001**.

Deux sujets voisins ne sont pas repris ici : le fonctionnement du rootless et le
décalage d'UID sont traités dans le lab `l4-podman-basic`, la gestion des images
dans `l4-podman-images`. On part du principe que `podman run` vous est acquis.
Podman n'étant pas installé sur une AlmaLinux minimale, tout commence par
`command -v podman || sudo dnf -y install podman`.

### Pourquoi Quadlet, et plus `podman generate systemd`

L'ancienne méthode consistait à lancer le conteneur à la main, puis à demander à
Podman d'en déduire une unité systemd. Elle existe toujours, et elle vous
prévient :

```bash
podman run -d --name ancienne-methode docker.io/library/alpine:3.21 sleep 900
podman generate systemd --name ancienne-methode
```

```text
DEPRECATED command:
It is recommended to use Quadlets for running containers and pods under systemd.
[...]
ExecStart=/usr/bin/podman start ancienne-methode
PIDFile=/run/user/1001/containers/overlay-containers/8cba58e44a0cdc30b09[...]/userdata/conmon.pid
Type=forking
```

Regardez le `PIDFile` : il contient l'**identifiant du conteneur existant**.
L'unité produite ne décrit pas une intention, elle photographie un conteneur
déjà créé, et changer un port ou un volume impose de tout refaire. C'est ce que
le guide résume par « complètes mais figées ». Quadlet inverse la logique : vous
écrivez un fichier déclaratif, et un **générateur systemd** en fabrique l'unité
à chaque `daemon-reload` et à chaque boot.

### Écrire l'unité `.container` et regarder ce que systemd en fait

En rootless, les fichiers vivent dans `~/.config/containers/systemd/`, qui
n'existe pas par défaut :

```bash
mkdir -p ~/.config/containers/systemd
cat > ~/.config/containers/systemd/veilleuse.container << 'EOF'
[Unit]
Description=Veilleuse : horodatage toutes les 30 s

[Container]
Image=docker.io/library/alpine:3.21
ContainerName=veilleuse
Exec=/bin/sh -c "while true; do date -Iseconds; sleep 30; done"

[Service]
Restart=always

[Install]
WantedBy=default.target
EOF
```

Quatre sections, quatre rôles : `[Unit]` pour les métadonnées et les
dépendances, `[Container]` pour ce que ferait `podman run`, `[Service]` pour le
comportement systemd, `[Install]` pour l'accrochage au démarrage.

À ce stade, systemd ignore encore tout de ce fichier :

```bash
systemctl --user list-unit-files 'veilleuse*'    # 0 unit files listed.
ls /run/user/1001/systemd/generator/             # No such file or directory
```

C'est le `daemon-reload` qui déclenche le générateur :

```bash
systemctl --user daemon-reload
ls -l /run/user/1001/systemd/generator/
systemctl --user list-unit-files 'veilleuse*'
```

```text
drwxr-xr-x. 2 ansible ansible  60 Jul 22 16:16 default.target.wants
-rw-r--r--. 1 ansible ansible 961 Jul 22 16:16 veilleuse.service

UNIT FILE         STATE     PRESET
veilleuse.service generated -
```

**L'unité n'est pas dans `~/.config/systemd/user/`, elle est sous `/run`**, donc
en mémoire, recréée à chaque reload. D'où son état : `generated`. Pour la lire :

```bash
systemctl --user cat veilleuse.service
```

```text
# /run/user/1001/systemd/generator/veilleuse.service
# Automatically generated by /usr/lib/systemd/user-generators/podman-user-generator
SourcePath=/home/ansible/.config/containers/systemd/veilleuse.container
[...]
ExecStop=/usr/bin/podman rm -v -f -i veilleuse
Type=notify
ExecStart=/usr/bin/podman run --name veilleuse --replace --rm --cgroups=split \
  --sdnotify=conmon -d docker.io/library/alpine:3.21 /bin/sh -c "while[...]"
[Install]
WantedBy=default.target
```

Le `podman run` que vous n'avez pas écrit est là, avec `--replace` (le nom est
repris même s'il traîne un ancien conteneur) et `--rm` (rien ne subsiste après
l'arrêt). `SourcePath=` désigne votre fichier : c'est lui la source de vérité.

Il ne reste qu'à démarrer :

```bash
systemctl --user start veilleuse.service
systemctl --user status veilleuse.service
```

```text
● veilleuse.service - Veilleuse : horodatage toutes les 30 s
     Loaded: loaded (/home/ansible/.config/containers/systemd/veilleuse.container; generated)
     Active: active (running) since Wed 2026-07-22 16:16:43 UTC; 3s ago
   Main PID: 3536 (conmon)
     CGroup: /user.slice/user-1001.slice/user@1001.service/app.slice/veilleuse.service
             ├─3539 /bin/sh -c "while true; do date -Iseconds; sleep 30; done"
```

`Loaded:` pointe vers le `.container`, pas vers le `.service` : systemd sait
d'où vient l'unité. Le conteneur, lui, s'appelle `veilleuse` parce que
`ContainerName=` l'impose. Sans cette directive, Quadlet **préfixe le nom du
fichier par `systemd-`**, ce qu'un second fichier `balise.container` confirme :

```text
NAMES           STATUS
systemd-balise  Up 2 seconds
veilleuse       Up 2 seconds
```

Retenez ce préfixe : c'est ce nom que les autres conteneurs devront viser pour
joindre celui-ci sur un réseau Podman.

### Rootless ou rootful : deux répertoires, deux jeux de commandes

C'est la première source d'erreur. Le mécanisme est identique, tout le reste
diffère :

| | Rootless (votre compte) | Rootful (système) |
|---|---|---|
| Répertoire | `~/.config/containers/systemd/` | `/etc/containers/systemd/` |
| Recharger | `systemctl --user daemon-reload` | `sudo systemctl daemon-reload` |
| Démarrer | `systemctl --user start <nom>` | `sudo systemctl start <nom>` |
| Unité générée | `/run/user/<uid>/systemd/generator/` | `/run/systemd/generator/` |
| `[Install] WantedBy=` | `default.target` | `multi-user.target` |
| `enable-linger` | **indispensable** | sans objet |

Le répertoire système existe dès l'installation, vide, livré par le paquet
`containers-common` (`rpm -qf /etc/containers/systemd/` le confirme) ; le
répertoire rootless, lui, est à créer. Une unité rootful tourne sous l'identité
root et peut publier un port sous 1024 ; une unité rootless reste dans votre
compte et n'a besoin d'aucun privilège. Le guide conseille de commencer en
rootless et de ne basculer que sur contrainte technique (port privilégié,
matériel, service partagé).

### Le piège : `systemctl enable` ne s'applique pas à une unité générée

Le réflexe systemd habituel échoue, et le message dit exactement pourquoi :

```bash
systemctl --user enable veilleuse.service
```

```text
Failed to enable unit: Unit /run/user/1001/systemd/generator/veilleuse.service is
transient or generated
```

`systemctl enable` fabrique un lien symbolique vers un fichier d'unité. Ici il
n'y a pas de fichier à lier : l'unité est recréée sous `/run` à chaque reload.
C'est le `[Install] WantedBy=` **du fichier `.container`** qui fait le travail,
et le générateur pose lui-même le lien. Preuve en retirant la section, puis en
la remettant, `daemon-reload` entre les deux :

```bash
ls -l /run/user/1001/systemd/generator/default.target.wants/
```

```text
# sans [Install]
ls: cannot access '[...]/default.target.wants/': No such file or directory

# avec [Install]
lrwxrwxrwx. 1 ansible ansible 20 Jul 22 16:17 veilleuse.service -> ../veilleuse.service
```

Sans `[Install]`, l'unité existe et se démarre à la main, mais rien ne la
réclame au boot. `systemctl --user is-enabled veilleuse.service` répond
`generated`, jamais `enabled` : ne cherchez pas cet état, il ne viendra pas.

### La preuve par le redémarrage

Une seule vérification compte : couper la machine et regarder. Avant reboot, le
service est `active`, le conteneur `Up`, et `loginctl list-users` donne
`1001 ansible no active` (colonne LINGER à `no`). Après `sudo systemctl reboot`,
on se reconnecte **avec un autre compte**, pour ne surtout pas ouvrir de session
à l'utilisateur propriétaire du service :

```bash
systemctl is-active user@1001.service
pgrep -u ansible -a conmon || echo "aucun conmon pour ansible"
```

```text
inactive
aucun conmon pour ansible
```

Le fichier est pourtant toujours là, avec son `[Install]`. Ce qui manque, c'est
le **gestionnaire de session utilisateur** : sans session ouverte, systemd ne
lance pas `user@1001.service`, donc personne n'atteint `default.target`, donc
rien ne démarre. C'est le pendant exact du conteneur rootless tué à la
déconnexion (`Exited (137)`, voir `l4-podman-basic`). Le correctif est le même :

```bash
loginctl enable-linger        # ou : sudo loginctl enable-linger <user>
ls -l /var/lib/systemd/linger/
```

```text
-rw-r--r--. 1 root root 0 Jul 22 16:18 ansible
```

Un fichier vide au nom de l'utilisateur : c'est tout ce que `enable-linger`
écrit, et c'est ce qui survit au reboot. Second essai, même protocole, même
observateur :

```text
 UID USER    LINGER STATE
1001 ansible yes    lingering

systemctl is-active user@1001.service     → active
pgrep -u ansible -a conmon                → 1181 /usr/bin/conmon [...] -n veilleuse
systemctl --user is-active veilleuse      → active
```

Et l'horodatage tranche la question de savoir si le service a démarré au boot ou
à la connexion :

```bash
uptime -s                                            # 2026-07-22 16:18:53
systemctl --user show veilleuse.service -p ActiveEnterTimestamp
```

```text
ActiveEnterTimestamp=Wed 2026-07-22 16:19:00 UTC
```

Sept secondes après le démarrage de la machine, sans qu'aucune session n'ait été
ouverte. **En rootless, `[Install]` sans `enable-linger` ne persiste pas au
boot** : lors du premier essai, le service est bien remonté, mais seulement à
l'instant où l'utilisateur a ouvert sa session ssh (`Up 3 seconds` juste après
la connexion). Un service qui attend qu'on se connecte n'est pas un service.

### Ce que systemd apporte en plus

`Restart=always` couvre le crash comme le reboot. On tue le conteneur par en
dessous, dans le dos de systemd :

```bash
podman kill veilleuse ; sleep 6
systemctl --user is-active veilleuse.service
systemctl --user show veilleuse.service -p NRestarts
```

```text
active
veilleuse Up 5 seconds
NRestarts=1
```

Symétriquement, `systemctl --user stop veilleuse.service` puis `podman ps -a` ne
rend **aucune ligne** : le `--rm` de la commande générée ne laisse pas de
cadavre `Exited` derrière lui, contrairement à un `podman run` manuel.

Une réserve sur les journaux : le guide propose `journalctl --user -u <nom>`.
Sur cette VM, la commande répond `No journal files were found`, car
`/run/log/journal/` ne contient que des fichiers `system*.journal`, sans journal
par utilisateur. Les messages sont bien là, mais dans le journal système
(`sudo journalctl -u user@1001.service`), ou via `podman logs` :

```text
Jul 22 16:16:43 atelier systemd[2137]: Started veilleuse.service - Veilleuse [...]
Jul 22 16:16:43 atelier veilleuse[3536]: 2026-07-22T16:16:43+00:00
```

### Quand rien n'apparaît : dépanner un Quadlet

Le mode d'échec le plus déroutant : un fichier au contenu invalide ne produit
**pas d'unité du tout**. Avec une directive mal orthographiée (`Imaje=`), après
`daemon-reload`, `systemctl --user start cassee.service` répond :

```text
Failed to start cassee.service: Unit cassee.service not found.
```

Aucune erreur au `daemon-reload`, aucune trace. Il faut appeler le générateur à
la main, avec `/usr/libexec/podman/quadlet --dryrun --user`, pour obtenir le
diagnostic :

```text
quadlet-generator: converting "cassee.container": unsupported key 'Imaje' in group
'Container' in /home/ansible/.config/containers/systemd/cassee.container
quadlet-generator: processing encountered some errors
```

D'où l'ordre de dépannage, du plus fréquent au plus rare :

1. le fichier est-il dans le **bon répertoire** pour le mode choisi, avec
   l'extension `.container`, et le `daemon-reload` a-t-il été fait avec ou sans
   `--user` selon le mode ;
2. `systemctl [--user] list-unit-files '<nom>*'` : l'unité existe-t-elle, et
   est-elle bien à l'état `generated` ;
3. sinon, `/usr/libexec/podman/quadlet --dryrun [--user]` donne la ligne fautive ;
4. si l'unité existe mais échoue au démarrage, lisez le journal (image
   introuvable, port déjà pris, refus de permission).

Volumes, réseaux, dépendances entre services et durcissement sont hors sujet
ici : le [guide compagnon](https://blog.stephane-robert.info/docs/conteneurs/moteurs-conteneurs/podman/quadlet/)
les traite recette par recette.
