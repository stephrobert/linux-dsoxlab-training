# Lab — boot-persistent container with Quadlet

## Reminder

[**Quadlet on the companion guide**](https://blog.stephane-robert.info/docs/conteneurs/moteurs-conteneurs/podman/quadlet/)

Quadlet turns the `*.container` files dropped into a directory known to systemd
into services, at `daemon-reload`. A `[Container]` section describes the image
and the command; the `[Install] WantedBy=` section makes the generated service
start at boot. The service name is the file name, followed by `.service`.

## The course

The examples below set up a service named `veilleuse`, in **rootless** mode,
from the `docker.io/library/alpine:3.21` image: the challenge will ask for
another name, another image and the other mode. The point is to learn the move
and to know how to transpose it, not to copy a line. Every output comes from an
**AlmaLinux 10.2** VM running **Podman 5.8.2**, user `ansible` with **UID
1001**.

Two neighbouring subjects are not covered here: how rootless works and the UID
shift are handled in the `l4-podman-basic` lab, image management in
`l4-podman-images`. We assume you already have `podman run`. As Podman is not
installed on a minimal AlmaLinux, everything starts with
`command -v podman || sudo dnf -y install podman`.

### Why Quadlet, and no longer `podman generate systemd`

The old method consisted in starting the container by hand, then asking Podman
to derive a systemd unit from it. It still exists, and it warns you:

```bash
podman run -d --name ancienne-methode docker.io/library/alpine:3.21 sleep 900
podman generate systemd --name ancienne-methode
```

```text
DEPRECATED command:
It is recommended to use Quadlets for running containers and pods under systemd.
[...]
ExecStart=/usr/bin/podman start ancienne-methode
PIDFile=/run/user/1001/containers/overlay-containers/8cba58e44a0cdc30b09[...]/userdata/conmon.pid
Type=forking
```

Look at the `PIDFile`: it contains the **identifier of the existing container**.
The produced unit does not describe an intent, it photographs a container that
already exists, and changing a port or a volume means redoing everything. This
is what the guide sums up as "complete but frozen". Quadlet reverses the logic:
you write a declarative file, and a **systemd generator** builds the unit from
it at every `daemon-reload` and at every boot.

### Writing the `.container` unit and looking at what systemd does with it

In rootless mode, the files live in `~/.config/containers/systemd/`, which does
not exist by default:

```bash
mkdir -p ~/.config/containers/systemd
cat > ~/.config/containers/systemd/veilleuse.container << 'EOF'
[Unit]
Description=Veilleuse : horodatage toutes les 30 s

[Container]
Image=docker.io/library/alpine:3.21
ContainerName=veilleuse
Exec=/bin/sh -c "while true; do date -Iseconds; sleep 30; done"

[Service]
Restart=always

[Install]
WantedBy=default.target
EOF
```

Four sections, four roles: `[Unit]` for metadata and dependencies,
`[Container]` for what `podman run` would do, `[Service]` for the systemd
behaviour, `[Install]` for hooking into startup.

At this stage, systemd still knows nothing about this file:

```bash
systemctl --user list-unit-files 'veilleuse*'    # 0 unit files listed.
ls /run/user/1001/systemd/generator/             # No such file or directory
```

It is the `daemon-reload` that triggers the generator:

```bash
systemctl --user daemon-reload
ls -l /run/user/1001/systemd/generator/
systemctl --user list-unit-files 'veilleuse*'
```

```text
drwxr-xr-x. 2 ansible ansible  60 Jul 22 16:16 default.target.wants
-rw-r--r--. 1 ansible ansible 961 Jul 22 16:16 veilleuse.service

UNIT FILE         STATE     PRESET
veilleuse.service generated -
```

**The unit is not in `~/.config/systemd/user/`, it is under `/run`**, so in
memory, recreated at every reload. Hence its state: `generated`. To read it:

```bash
systemctl --user cat veilleuse.service
```

```text
# /run/user/1001/systemd/generator/veilleuse.service
# Automatically generated by /usr/lib/systemd/user-generators/podman-user-generator
SourcePath=/home/ansible/.config/containers/systemd/veilleuse.container
[...]
ExecStop=/usr/bin/podman rm -v -f -i veilleuse
Type=notify
ExecStart=/usr/bin/podman run --name veilleuse --replace --rm --cgroups=split \
  --sdnotify=conmon -d docker.io/library/alpine:3.21 /bin/sh -c "while[...]"
[Install]
WantedBy=default.target
```

The `podman run` you did not write is there, with `--replace` (the name is taken
over even if an old container is lying around) and `--rm` (nothing survives the
stop). `SourcePath=` designates your file: that is the source of truth.

All that is left is to start it:

```bash
systemctl --user start veilleuse.service
systemctl --user status veilleuse.service
```

```text
● veilleuse.service - Veilleuse : horodatage toutes les 30 s
     Loaded: loaded (/home/ansible/.config/containers/systemd/veilleuse.container; generated)
     Active: active (running) since Wed 2026-07-22 16:16:43 UTC; 3s ago
   Main PID: 3536 (conmon)
     CGroup: /user.slice/user-1001.slice/user@1001.service/app.slice/veilleuse.service
             ├─3539 /bin/sh -c "while true; do date -Iseconds; sleep 30; done"
```

`Loaded:` points to the `.container`, not to the `.service`: systemd knows where
the unit comes from. The container itself is called `veilleuse` because
`ContainerName=` imposes it. Without that directive, Quadlet **prefixes the file
name with `systemd-`**, which a second file `balise.container` confirms:

```text
NAMES           STATUS
systemd-balise  Up 2 seconds
veilleuse       Up 2 seconds
```

Remember that prefix: this is the name other containers will have to target to
reach this one over a Podman network.

### Rootless or rootful: two directories, two sets of commands

This is the first source of error. The mechanism is identical, everything else
differs:

| | Rootless (your account) | Rootful (system) |
|---|---|---|
| Directory | `~/.config/containers/systemd/` | `/etc/containers/systemd/` |
| Reload | `systemctl --user daemon-reload` | `sudo systemctl daemon-reload` |
| Start | `systemctl --user start <name>` | `sudo systemctl start <name>` |
| Generated unit | `/run/user/<uid>/systemd/generator/` | `/run/systemd/generator/` |
| `[Install] WantedBy=` | `default.target` | `multi-user.target` |
| `enable-linger` | **indispensable** | not applicable |

The system directory exists from installation onwards, empty, shipped by the
`containers-common` package (`rpm -qf /etc/containers/systemd/` confirms it);
the rootless directory, on the other hand, has to be created. A rootful unit
runs under the root identity and can publish a port below 1024; a rootless unit
stays inside your account and needs no privilege. The guide advises starting
rootless and switching only under a technical constraint (privileged port,
hardware, shared service).

### The trap: `systemctl enable` does not apply to a generated unit

The usual systemd reflex fails, and the message says exactly why:

```bash
systemctl --user enable veilleuse.service
```

```text
Failed to enable unit: Unit /run/user/1001/systemd/generator/veilleuse.service is
transient or generated
```

`systemctl enable` builds a symbolic link to a unit file. Here there is no file
to link to: the unit is recreated under `/run` at every reload. It is the
`[Install] WantedBy=` **of the `.container` file** that does the job, and the
generator lays down the link itself. Proof by removing the section, then putting
it back, with a `daemon-reload` in between:

```bash
ls -l /run/user/1001/systemd/generator/default.target.wants/
```

```text
# without [Install]
ls: cannot access '[...]/default.target.wants/': No such file or directory

# with [Install]
lrwxrwxrwx. 1 ansible ansible 20 Jul 22 16:17 veilleuse.service -> ../veilleuse.service
```

Without `[Install]`, the unit exists and can be started by hand, but nothing
requires it at boot. `systemctl --user is-enabled veilleuse.service` answers
`generated`, never `enabled`: do not look for that state, it will not come.

### The proof by reboot

Only one check counts: cut the machine off and look. Before the reboot, the
service is `active`, the container `Up`, and `loginctl list-users` gives
`1001 ansible no active` (LINGER column at `no`). After `sudo systemctl reboot`,
you log back in **with another account**, so as not to open a session for the
user who owns the service:

```bash
systemctl is-active user@1001.service
pgrep -u ansible -a conmon || echo "aucun conmon pour ansible"
```

```text
inactive
aucun conmon pour ansible
```

The file is still there, though, with its `[Install]`. What is missing is the
**user session manager**: with no session open, systemd does not start
`user@1001.service`, so nobody reaches `default.target`, so nothing starts. This
is the exact counterpart of the rootless container killed on logout
(`Exited (137)`, see `l4-podman-basic`). The fix is the same:

```bash
loginctl enable-linger        # or: sudo loginctl enable-linger <user>
ls -l /var/lib/systemd/linger/
```

```text
-rw-r--r--. 1 root root 0 Jul 22 16:18 ansible
```

An empty file named after the user: that is all `enable-linger` writes, and that
is what survives the reboot. Second attempt, same protocol, same observer:

```text
 UID USER    LINGER STATE
1001 ansible yes    lingering

systemctl is-active user@1001.service     → active
pgrep -u ansible -a conmon                → 1181 /usr/bin/conmon [...] -n veilleuse
systemctl --user is-active veilleuse      → active
```

And the timestamp settles the question of whether the service started at boot or
at login:

```bash
uptime -s                                            # 2026-07-22 16:18:53
systemctl --user show veilleuse.service -p ActiveEnterTimestamp
```

```text
ActiveEnterTimestamp=Wed 2026-07-22 16:19:00 UTC
```

Seven seconds after the machine started, without any session having been opened.
**In rootless mode, `[Install]` without `enable-linger` does not persist across
a boot**: on the first attempt, the service did come back, but only at the
moment the user opened his ssh session (`Up 3 seconds` right after the
connection). A service that waits for someone to log in is not a service.

### What systemd brings on top

`Restart=always` covers the crash as well as the reboot. Kill the container from
underneath, behind systemd's back:

```bash
podman kill veilleuse ; sleep 6
systemctl --user is-active veilleuse.service
systemctl --user show veilleuse.service -p NRestarts
```

```text
active
veilleuse Up 5 seconds
NRestarts=1
```

Symmetrically, `systemctl --user stop veilleuse.service` then `podman ps -a`
return **no line**: the `--rm` of the generated command leaves no `Exited`
corpse behind, unlike a manual `podman run`.

One reservation about the journals: the guide suggests
`journalctl --user -u <name>`. On this VM, the command answers
`No journal files were found`, because `/run/log/journal/` only contains
`system*.journal` files, with no per-user journal. The messages are indeed
there, but in the system journal (`sudo journalctl -u user@1001.service`), or
through `podman logs`:

```text
Jul 22 16:16:43 atelier systemd[2137]: Started veilleuse.service - Veilleuse [...]
Jul 22 16:16:43 atelier veilleuse[3536]: 2026-07-22T16:16:43+00:00
```

### When nothing shows up: troubleshooting a Quadlet

The most disconcerting failure mode: a file with invalid content produces **no
unit at all**. With a misspelled directive (`Imaje=`), after `daemon-reload`,
`systemctl --user start cassee.service` answers:

```text
Failed to start cassee.service: Unit cassee.service not found.
```

No error at `daemon-reload`, no trace. You have to call the generator by hand,
with `/usr/libexec/podman/quadlet --dryrun --user`, to get the diagnosis:

```text
quadlet-generator: converting "cassee.container": unsupported key 'Imaje' in group
'Container' in /home/ansible/.config/containers/systemd/cassee.container
quadlet-generator: processing encountered some errors
```

Hence the troubleshooting order, from the most frequent to the rarest:

1. is the file in the **right directory** for the chosen mode, with the
   `.container` extension, and was the `daemon-reload` done with or without
   `--user` according to the mode;
2. `systemctl [--user] list-unit-files '<name>*'`: does the unit exist, and is
   it indeed in the `generated` state;
3. otherwise, `/usr/libexec/podman/quadlet --dryrun [--user]` gives the faulty
   line;
4. if the unit exists but fails to start, read the journal (image not found,
   port already taken, permission denied).

Volumes, networks, dependencies between services and hardening are out of scope
here: the [companion guide](https://blog.stephane-robert.info/docs/conteneurs/moteurs-conteneurs/podman/quadlet/)
covers them recipe by recipe.
